# ADR 008 — OpenAPI 3.1 as the single source of truth

- **Status:** accepted
- **Date:** 2026-04-29

## Context

We have an HTTP API documented in three places:
- [docs.liqaa.io](https://liqaa.io/docs)
- The Console UI's reference tab
- Each SDK's typings (TS, PHP, Python, Go)

Without a single source, these will drift. We've all seen APIs where the docs, the SDK, and the actual HTTP behaviour disagree. That's the fastest way to lose developer trust.

## Decision

The repo [liqaa-openapi](https://github.com/hartemyaakoub/liqaa-openapi) holds **`openapi.yaml`** — an OpenAPI 3.1 document that is the **only** description of our API. Everything else is generated or derived from it:

- Reference page at [liqaa.io/reference](https://liqaa.io/reference) (Redoc)
- TypeScript types in `@liqaa/js`
- PHP request/response classes (planned)
- Python `pydantic` models (planned)
- Go structs (planned)
- Postman collection (auto-published)

Changes are **PR-reviewed in `liqaa-openapi` first**. The Laravel routes file is verified against the spec on CI.

## Consequences

### Positive
- One place to look. One place to PR-review API changes.
- Backwards-compatibility breaks are visible in the diff.
- 50+ language clients can be generated by users via [openapi-generator](https://openapi-generator.tech).
- AI agents (Claude, GPT, Cursor) can consume the spec directly.

### Negative
- The HTTP server itself (Laravel) is *not* generated from the spec. They could drift.
- We mitigate by writing **contract tests** that fire every documented endpoint with valid + invalid payloads and verify the responses match the spec.
